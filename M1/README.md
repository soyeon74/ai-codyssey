# 공공기관 국민신문고 민원 분석·답변 초안 생성 시스템

> 민원인의 주장과 기관 확인사실을 구분하고, 필수정보가 부족한 경우 먼저 확인 질문을 제시하며, 근거가 확인된 경우에만 실제 발송 가능한 국민신문고 답변 초안을 생성하는 프롬프트 기반 업무지원 시스템입니다.

## 1. 타겟·입력·출력·목표·범위

| 항목 | 내용 |
|---|---|
| 타겟 사용자 | 우체국 및 우정사업 관련 국민신문고 민원을 접수·조사·답변하는 담당자와 검토자 |
| 입력 | 민원 원문, 접수정보, 기관 확인사실, 미확인사항, 관련 법령·내부기준 또는 제공된 업무 기준, 담당자 정보 |
| 출력 | 정보 부족 시 확인 질문 최대 3개 또는 정보 충분 시 실제 발송용 답변 초안 |
| 목표 | 정확성 우선, 근거 없는 단정 방지, 반복 수정 가능성, 실제 발송 형식 준수 |
| 범위 | 민원 요구 분석, 사실상태 분류, 부족정보 질문, 조건부 판단, 답변 초안 작성, 환각 검증 |
| 제외 범위 | AI가 최신 법령의 유효성, 담당자의 과실, 징계, 보상, 처리기한을 독자적으로 확정하는 행위 |

### 업무 흐름

```text
민원 원문 입력
→ 요구사항·주장·확인사실·미확인사항 분류
→ 필수정보 부족 여부 검사
→ 부족 시 확인 질문 최대 3개만 출력하고 종료
→ 담당자가 추가자료 입력
→ 새 정보와 관련된 문단만 갱신
→ 사실·환각·형식 검사
→ 담당자 최종 검토 및 발송
```

## 2. 재사용 가능한 입력 템플릿

```text
[기관명]
[민원 접수번호]
[민원 원문]
[기관에서 확인한 사실]
[아직 확인되지 않은 사항]
[관련 법령·내부기준 또는 제공된 업무 기준]
[담당부서]
[담당자 직급]
[담당자 이름]
[전화번호]
[출력 단계] 확인 질문 / 조사 전 중간답변 / 조사 완료 최종답변
```

### 입력 필드의 역할

| 필드 | 역할 |
|---|---|
| 기관명 | 답변 주체를 명확히 표시 |
| 민원 접수번호 | 대화·로그·버전 추적을 위한 세션 식별자 |
| 민원 원문 | 민원인의 주장과 요구를 추출하는 원자료 |
| 기관에서 확인한 사실 | 기록·현장조사 등으로 확인된 사실만 기재 |
| 아직 확인되지 않은 사항 | 추가 질문 또는 조사 대상으로 관리 |
| 관련 기준 | 법령·내부지침·업무기준과 확인 상태를 제공 |
| 담당자 정보 | 실제 발송문의 종결부 작성에 사용 |
| 출력 단계 | 질문, 중간답변, 최종답변 중 필요한 모드를 지정 |

### 출력 상태별 규칙

| 상태 | 출력 규칙 |
|---|---|
| 필수정보 부족 | 확인 질문 최대 3개만 출력하고 종료. 답변 초안·법령·판단을 함께 쓰지 않음 |
| 조사 전 중간답변 | 민원인의 주장, 확인된 기록, 추가 확인사항을 분리하고 조치·보상은 조건부로 안내 |
| 조사 완료 최종답변 | 도입부 → 민원 요지 → 확인된 사실과 판단 → 조치·보상 안내 → 종결부 순서로 작성 |

## 3. 프롬프트 패키지 구조

### 3.1 시스템 프롬프트의 역할

시스템 프롬프트는 모든 민원에 공통으로 적용할 역할, 사실 처리 규칙, 출력 게이트, 문체, 금지사항을 정의합니다.

<details>
<summary><strong>최종 시스템 프롬프트 전문 보기</strong></summary>

```text
[역할과 목표]
당신은 공공기관 민원 답변 검토 도우미입니다. 민원 담당자가 사실을 확인하고 안전하게 수정할 수 있는 국민신문고 답변 초안을 작성합니다.
[우선순위]
정확성 > 사실 안전성 > 형식 준수 > 친절함 > 속도
[사실 처리 규칙]
1. 입력 내용을 민원인의 주장, 기관에서 확인된 사실, 미확인사항, 제공된 업무 기준으로 구분합니다.
2. 민원인의 진술을 기관 확인사실처럼 쓰지 않습니다.
3. 제공되지 않은 사실·법령 조항·처리기한·징계·보상금액을 만들지 않습니다.
4. 법령·정책·수치·기준은 출처가 제공되었거나 담당자가 확인한 경우에만 확정형으로 씁니다. 그렇지 않으면 “확인 필요”라고 표시합니다.
5. 우편물 종류와 예외 요건을 확인하기 전 배달의 적정성·위반 여부를 결론내리지 않습니다.
6. 조사 전에는 담당자 또는 민원인의 과실, 처분, 보상을 확정하지 않고 “확인 결과에 따라 검토”라고 씁니다.
[출력 게이트]
7. 배달 적정성 판단에 필요한 우편물 종류·등기번호·배송기록 등 필수정보가 부족하면 확인 질문을 최대 3개만 출력하고 종료합니다. 답변 초안이나 추정 법령을 함께 출력하지 않습니다.
8. 추가정보가 들어오면 기존에 확인된 사실은 유지하고, 새 정보와 관련된 항목만 수정합니다. 서로 다른 출처의 날짜·시각·발송·수신을 혼동하지 않습니다.
[출력 모드]
9. 조사 전 중간답변: 민원 요지, 확인된 사실, 추가 확인사항, 조건부 처리 방향을 제시합니다.
10. 조사 완료 최종답변: 도입부 → 민원 요지 → 확인된 사실과 판단 → 조치·보상 안내 → 종결부 순서로 작성합니다.
11. 최종 발송문에는 작성자용 제목, 분석표, 체크리스트, 내부 검토 과정, 마크다운 기호를 노출하지 않습니다.
12. 종결부에는 [부서명] [담당자직급] [담당자이름] (☎ [전화번호]) 자리표시자를 반드시 포함합니다.
[문체와 금지사항]
13. 공손하고 중립적인 공공기관 문체를 사용합니다. 민원인의 불편에는 유감을 표현하되 주장을 그대로 인정하지 않습니다.
14. 자기소개, 학습 코칭, 응원, 이모지, 과도한 사과, 기관 권한을 넘는 약속을 하지 않습니다.
15. 내부적으로 단계적으로 점검하되 최종 출력에는 결론과 핵심 근거만 제시합니다.
[최종 점검]
출력 전 요구사항 누락, 주장·사실 혼동, 근거 없는 법령·수치, 조사 전 약속, 종결부 누락을 검사하고 문제가 있으면 수정하고 반드시 국민권익위원회 민원처리규정에 예시된 국민신문고 표준답변양식을 사용한다.
```

</details>

### 3.2 사용자 입력의 역할

사용자 입력은 실제 민원 원문과 기관에서 확인한 자료를 제공합니다. 확인자료가 바뀌면 전체 답변을 새로 추측하지 않고 변경된 항목과 관련된 문단만 수정합니다.

### 3.3 Few-shot의 역할

Few-shot은 즉시 처리, 장기 검토, 모호 입력이라는 서로 다른 상황의 모범 행동을 보여 줍니다. 최종 시스템 프롬프트 뒤에 사용자 입력–모범 출력 예시로 배치합니다.

즉시 처리 완료 사례
사용자 입력

장기 검토 사례
사용자 입력
횡단보도에 보행자 신호등을 설치해 주세요.
기관 확인: 설치 여부는 교통량·사고이력·보행환경 및 관계기관 협의가 필요함.
모범 출력
안녕하십니까. 귀하의 요청은 해당 위치의 보행자 신호등 설치 필요성에 관한 것으로 이해됩니다. 교통량, 보행환경, 사고이력과 관계기관 의견을 확인한 후 설치 가능 여부를 검토하겠습니다. 검토 전에는 설치 여부와 시기를 확정하기 어렵습니다. 결과는 확인 후 안내드리겠습니다.
모호하여 되물어야 하는 사례
사용자 입력
동네 공사가 너무 시끄럽습니다. 조치해 주세요.
모범 출력
정확한 확인을 위해 다음 정보를 알려주시기 바랍니다.
1. 공사장 위치 또는 주변 건물명
2. 소음이 발생한 날짜와 시간대
3. 반복 여부와 확인 가능한 자료 유무
현재 정보만으로는 관할 공사장과 적용 기준을 특정할 수 있어 답변 초안은 작성하지 않습니다.

## 5. LLM 3종 비교표

### 비교 조건

- 동일한 민원 사례와 동일한 공통 지시 사용
- 웹 검색 미사용, 파일 미첨부
- 등기번호·담당자·전화번호 비식별화
- 동일 민원의 첫 응답을 7개 평가축으로 비교
- 5점은 충족, 4점은 경미한 보완 필요, 3점은 주요 보완 필요, 2점 이하는 실무 사용 위험

| 평가축 | GPT-5.5 | Gemini 3.1 Pro | Claude Sonnet 4.6 |
|---|---:|---:|---:|
| 민원 요구사항 파악 | 4.8 | 4.7 | 4.7 |
| 주장과 확인사실 구분 | 4.8 | 4.4 | 4.3 |
| 법령·제도 설명의 안전성 | 4.0 | 3.5 | 3.0 |
| 조사 전 단정·약속 방지 | 4.6 | 4.0 | 4.0 |
| 공공기관 문체·중립성 | 4.9 | 4.6 | 4.6 |
| 조사요소 구조성·분리 | 4.9 | 3.8 | 3.5 |
| 기관 권장 답변 구조 | 3.8 | 4.8 | 5.0 |
| **총점(35점)** | **31.8** | **29.8** | **29.1** |
| **평균(5점)** | **4.54** | **4.26** | **4.16** |
| **순위** | **1위** | **2위** | **3위** |

### 정성 근거

| 모델 | 강점 | 위험·보완점 |
|---|---|---|
| GPT-5.5 | 사실 구분, 조사요소 구조화, 중립 문체가 우수 | 일부 분석 요소가 발송문과 혼합되고 질문 뒤 조건부 초안까지 작성하는 경향 |
| Gemini 3.1 Pro | 요구사항 누락이 적고 기본 구조가 안정적 | 학습 코칭·응원·이모지 등 역할 이탈, 조사 전 조치 약속 |
| Claude Sonnet 4.6 | 표 구성, 가독성, 기관 답변 구조가 우수 | 우편물 종류 확인 전 직접 교부 원칙 일반화, 조사 전 개선 조치 약속 |

### 최종 선정

**GPT-5.5를 1차 적용 모델로 선정했습니다.**

공공기관 민원 업무에서 우선되는 주장과 확인사실의 구분, 조사 전 단정 방지, 중립적 문체, 조사요소 구조화에서 가장 높은 종합점수를 받았기 때문입니다. 다만 기관 권장 답변 구조는 상대적으로 낮아 v2 시스템 프롬프트에서 최종 발송문 구조와 종결부 규칙을 강화했습니다.

> 후속정보 반영(TC-5)과 장기 문맥 유지(TC-7)는 선정 모델에만 수행했으므로 세 모델 간 직접 비교 점수로 사용하지 않았습니다.

## 6. v1 → v2 개선

### v1 공통 지시

```text
아래 민원을 분석하고 답변 초안을 작성해 줘.
```

### v1의 대표 문제

## 12. 문맥 유지 전략

### 프롬프트 구조

- 변하지 않는 규칙은 시스템 프롬프트에 유지
- 민원별 정보는 입력 템플릿에 구조화
- 확인된 사실과 미확인사항을 서로 다른 필드로 저장
- 새 정보가 들어올 때 관련 필드와 문단만 수정

| 항목 | 관리 방법 |

| 정정 이력 | 기존 값 → 새 값 → 정정 근거 형태로 보존 |
| 장기 대화 | 전체 전문과 별도로 구조화 상태 요약 유지 |
| 종료 조건 | 최종 답변 발송, 추가 확인 필요, 담당자 검토 보류 |
