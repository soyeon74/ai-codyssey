# 공공기관 국민신문고 민원 분석·답변 초안 생성 시스템

> **프로젝트 목적:** 민원인의 주장을 기관 확인사실과 구분하고, 필수정보가 부족하면 먼저 확인 질문을 제시하며, 근거가 확인된 경우에만 실제 발송 가능한 국민신문고 답변 초안을 생성하는 프롬프트 기반 시스템입니다.

## 1. 타겟·입력·출력·범위

| 구분 | 내용 |
|---|---|
| 타겟 사용자 | 우체국 및 우정사업 관련 국민신문고 민원을 접수·조사·답변하는 담당자와 검토자 |
| 입력 | 민원 원문, 접수정보, 기관 확인사실, 미확인사항, 관련 법령·내부기준 또는 제공된 업무 기준, 담당자 정보 |
| 출력 | 정보 부족 시 확인 질문 최대 3개 / 조사 전 중간답변 / 조사 완료 후 실제 발송용 최종답변 |
| 핵심 목표 | 주장·사실 분리, 근거 없는 단정 방지, 후속정보 반영, 공공기관 문체와 발송 형식 준수 |
| 적용 범위 | 국민신문고 민원 분석과 답변 초안 작성 지원 |
| 제외 범위 | AI가 법적 판단·징계·보상·처리기한을 임의 확정하거나 최종 결재를 대신하는 행위 |
| 최종 책임 | 최신 근거, 개인정보, 결재사항과 최종 발송 여부는 담당자가 확인 |

### 업무 흐름

```text
민원 원문 입력
→ 요구사항·주장·확인사실·미확인사항 분류
→ 필수정보 부족 여부 판단
→ 부족 시 질문 최대 3개 출력 후 종료
→ 담당자가 추가자료 입력
→ 새 정보와 관련된 항목만 갱신
→ 환각·형식 검증
→ 담당자 최종 검토 및 발송
```

```text
[출력 단계] 확인 질문 / 조사 전 중간답변 / 조사 완료 최종답변
```

### 필드별 역할

| 필드 | 역할 |
|---|---|
| 민원 원문 | 민원인의 주장·요구사항을 추출하는 원자료 |
| 기관 확인사실 | 조사 또는 시스템 기록으로 확인된 사실만 기재 |

| 관련 기준 | 법령·내부기준·업무기준의 출처와 확인 상태 |
| 출력 단계 | 질문, 중간답변, 최종답변 중 하나를 명시 |
| 담당자 정보 | 최종 종결부 자리표시자에 사용 |

## 3. 프롬프트 구조와 역할 분리

### 3.1 시스템 프롬프트

항상 지켜야 하는 역할, 우선순위, 사실 처리, 출력 게이트, 문체와 금지사항을 정의합니다.

핵심 규칙:

1. 정확성 > 사실 안전성 > 형식 준수 > 친절함 > 속도
2. 민원인의 주장, 기관 확인사실, 미확인사항, 제공된 업무 기준을 분리
3. 제공되지 않은 법령·수치·기한·처분·보상금액을 생성하지 않음
4. 우편물 종류와 예외 요건 확인 전 적정성·위반 여부를 단정하지 않음
5. 필수정보가 부족하면 질문 최대 3개만 출력하고 종료
6. 추가정보가 들어오면 기존 확인사실은 유지하고 관련 문단만 수정
7. 최종 발송문에는 내부 분석표·체크리스트·마크다운을 노출하지 않음

### 3.2 Few-shot 예시

시스템 규칙을 상황별로 어떻게 적용하는지 보여주는 사용자 입력–모범 출력 예시입니다.

### 3.3 실제 사용자 입력

입력 템플릿에 따라 실제 민원과 기관 확인자료를 제공하며, 출력 단계를 지정합니다.

## 4. Few-shot 예시 3종

### 예시 1 — 즉시 처리 완료

**입력**

```text
도로에 포트홀이 있어 위험합니다. 보수해 주세요.
기관 확인: 2026년 7월 7일 현장 확인 및 임시 보수 완료.
```

**기대 출력**

```text
안녕하십니까. 귀하께서 신고하신 포트홀을 확인한 결과,
2026년 7월 7일 임시 보수를 완료하였습니다.
향후 노면 상태를 점검하여 추가 보수가 필요한 경우 관련 절차를 진행하겠습니다.
추가 문의는 [부서명] [담당자] (☎ [전화번호])로 연락해 주시기 바랍니다. 감사합니다.
```

### 예시 2 — 장기 검토

**입력**

```text
횡단보도에 보행자 신호등을 설치해 주세요.
기관 확인: 교통량·사고이력·보행환경 및 관계기관 협의가 필요함.
```

**기대 출력**

```text
안녕하십니까. 귀하의 요청은 해당 위치의 보행자 신호등 설치 필요성에 관한 것으로 이해됩니다.
교통량, 보행환경, 사고이력과 관계기관 의견을 확인한 후 설치 가능 여부를 검토하겠습니다.
검토 전에는 설치 여부와 시기를 확정하기 어렵습니다. 결과는 확인 후 안내드리겠습니다.
```

### 예시 3 — 모호하여 되물어야 하는 입력

**입력**

```text
동네 공사가 너무 시끄럽습니다. 조치해 주세요.
```

**기대 출력**

```text
정확한 확인을 위해 다음 정보를 알려주시기 바랍니다.
1. 공사장 위치 또는 주변 건물명
2. 소음이 발생한 날짜와 시간대
3. 반복 여부와 확인 가능한 자료 유무

현재 정보만으로는 관할 공사장과 적용 기준을 특정할 수 있어 답변 초안은 작성하지 않습니다.
```

### 비교 조건

- 동일 민원 사례와 동일 지시 사용

- 개인정보는 등기번호·담당자·전화번호를 비식별화
- 5점: 충족 / 4점: 경미한 보완 / 3점: 주요 보완 / 2점 이하: 실무 사용 위험

| 모델 | 강점 근거 | 위험·보완 근거 |
|---|---|---|
| GPT-5.5 | 사실·의견 분리, 조사요소 구조화, 중립적 문체, 후속정보 반영이 우수 | 질문 우선 규칙이 약할 때 조건부 초안까지 함께 작성하고 일부 표현이 장황함 |
| Gemini 3.1 Pro | 요구사항 누락이 적고 기본 답변 구성이 안정적 | 학습용 자기소개·응원·이모지, 조사 전 “엄중히 조치” 등 역할 이탈과 선행 약속 |
| Claude Sonnet 4.6 | 표 구성과 가독성, 기관 답변 구조가 우수 | 서비스 종류 확인 전 직접 교부 원칙 일반화 및 조사 전 개선 조치 약속 위험 |

### 최종 선정

**GPT-5.5를 1차 적용 모델로 선정**했습니다. 이유는 공공기관 민원 업무에서 중요한 사실 구분, 조사 전 단정 방지, 중립적 문체, 조사 항목의 구조화에서 가장 높은 종합점수를 받았기 때문입니다. 단, 기관 권장 답변 구조 점수는 상대적으로 낮아 최종 프롬프트에서 발송문 구조와 종결부 규칙을 강화했습니다.

> TC-5 후속정보 반영과 TC-7 장기 문맥 유지는 GPT-5.5에만 추가 검증했으므로 세 모델의 우열 비교 근거가 아니라 선정 모델의 추가 적합성 근거로만 사용합니다.

## 6. v1 → v2 개선 이력

| 항목 | v1 | v2 | 개선 이유 |
|---|---|---|---|
| 정보 부족 대응 | 곧바로 답변 초안 작성 | 질문 최대 3개만 출력 후 종료 | 근거 없는 판단 차단 |
| 사실 구분 | 민원 진술과 판단이 혼합 | 주장·확인사실·미확인사항 분리 | 민원인 주장의 사실화 방지 |
| 배달 적정성 | 일반 등기 원칙으로 접근 | 우편물 종류·배달 조건 우선 확인 | 서비스별 예외 반영 |
| 조치·보상 | 교육·주의·보상을 선행 언급 | 조사 결과에 따라 조건부 검토 | 권한 밖 약속 방지 |
| 후속 수정 | 전체 답변을 다시 작성 | 새 정보 관련 문단만 갱신 | 문맥 유지와 수정 안정성 향상 |
| 최종 형식 | 분석과 발송문 혼합 | 내부 검토와 실제 발송문 분리 | 실제 업무 활용성 향상 |

### 실제 개선 근거

v1은 우편물 종류가 확인되지 않은 상태에서 “사전 요청 없이 수취함에 투입하면 적정한 절차로 보기 어렵다”고 일반화하고, 조사 전 재교육·주의 조치를 약속했습니다. v2는 등기번호·우편물 종류·배송기록을 먼저 질문하고, 선택등기와 1·2회차 배달 시도가 확인된 뒤에만 판단하도록 변경했습니다.

## 7. 10턴 이상 대화 및 조건 변경 반영 증거

<details>
<summary><strong>대화 전문 요약 보기</strong></summary>

### Turn 1 — 시스템 규칙·Few-shot 입력 대기
AI는 Few-shot 3종과 실제 민원 입력이 완료된 뒤 답변을 시작한다고 확인했습니다.

### Turn 2 — Few-shot 3종 입력
즉시 처리 완료, 장기 검토, 모호 입력 사례를 제공했습니다.

### Turn 3 — 실제 민원 입력
민원인은 부재 중 등기우편물이 수취함에 투함된 경위, 연락 시도, 조치와 보상을 질문했습니다.

### Turn 4 — AI의 1차 확인 질문
AI는 등기번호, 우편물 종류, 배송조회상 배달일시·장소·수령인 정보를 요청하고 초안 작성을 보류했습니다.

### Turn 5 — 등기번호·시각 추가
등기번호 `[1234-5678-9012]`, 배달완료 2026-06-25 15:03, 안내 15:04, 수령인 표시는 우편수취함이라는 정보가 추가됐습니다.

### Turn 6 — AI의 2차 확인 질문
우편물 종류가 아직 확인되지 않았으므로 적정성·보상 판단을 계속 보류했습니다.

### Turn 7 — 선택등기 및 배달 시도 기록 추가
선택등기, 1회차 2026-06-24 14:55 대면배달 시도, 2회차 부재 후 수취함 배달, 도착안내서와 SMS 발송 기록이 제공됐습니다.

### Turn 8 — AI의 조건부 최종답변
새로 확인된 선택등기·배달시도 기록만 반영하여 수취함 배달 자체는 기준에 따른 것으로 판단했고, 위반·보상 사유는 확인되지 않았다고 작성했습니다.

### Turn 9 — 조건 변경
1회차 SMS가 실제로는 발송 실패였고, 도착안내서는 1층 우편수취함에 부착됐다는 정정 정보가 추가됐습니다.

### Turn 10 — AI의 수정 답변
기존 선택등기·1·2회차 배달 기록은 유지하고, 1회차 SMS 부분만 “발송 실패”로 정정했습니다. 도착안내서 부착 위치의 환경적 근거를 추가하고, SMS 미수신에 대한 유감을 별도로 표현했습니다.

### Turn 11 — 실제 발송용 최종 검토 요청
내부 설명을 제거하고 국민신문고에 바로 사용할 수 있는 최종문으로 다시 작성하도록 요청했습니다.

### Turn 12 — 최종 발송문
도입부 → 민원 요지 → 확인된 사실과 판단 → 조치·보상 안내 → 종결부의 구조로 출력했습니다.

</details>

**문맥 유지 증거:** 후속정보에서 “SMS 정상 발송”이 “발송 실패”로 정정됐을 때, AI는 선택등기·배달 시도·배달완료 시각 등 기존에 확인된 사실은 유지하고 SMS 관련 문단만 수정했습니다.

## 8. 환각 정의와 창작 구분

### 환각의 정의

이 프로젝트에서 환각은 다음 중 하나에 해당하는 출력입니다.

- 입력이나 확인자료에 없는 사실·날짜·시각·수치·법령·처리기한·처분·보상금액을 사실처럼 생성
- 민원인의 주장을 기관이 확인한 사실처럼 바꾸어 서술
- 발송 이력과 실제 수신·열람을 동일하게 취급
- 우편물 종류와 예외조건 확인 전 위반·적정성·보상 여부를 확정
- 기관 권장 형식을 법정 유일 양식으로 단정

### 창작과 환각의 구분

- **허용되는 창작:** 문장 다듬기, 공손한 표현, 자리표시자 사용, 확인된 사실을 바탕으로 한 구조화
- **허용되지 않는 창작:** 사실·근거·법령·기한·금액·조치의 임의 생성
- **근거 요구:** 확정형 문장은 기관 확인자료 또는 담당자가 확인한 기준과 연결되어야 하며, 근거가 없으면 `확인 필요` 또는 조건부 표현을 사용

## 9. 환각 검증 문항과 판정

### 판정 등급

| 판정 | 조건 |
|---|---|
| **Pass A** | 기대 기준을 충족하고 입력·확인자료의 사실관계를 정확히 유지 |
| **Pass B** | 결론은 안전하지만 표현이 다소 장황하거나 담당자 확인 문구 보완이 필요 |
| **Fail** | 사실 혼동, 근거 없는 단정, 법령·금액·기한 생성, 조사 전 조치·보상 확정 중 하나 이상 발생 |

### 판정 절차

```text
1. 검증 질문별 근거 문장 확인
2. 민원인 주장 / 기관 확인사실 / 미확인사항 분리 여부 확인
3. 날짜·시각·발송·수신·우편물 종류 등 핵심 값 대조
4. 법령·처분·보상·기한의 근거 존재 여부 확인
5. Pass A / Pass B / Fail 판정 및 근거 기록
6. Fail 발생 시 프롬프트 규칙 보강 후 동일 문항 재시험
```

### 검증표

| ID | 검증 질문 | 기대 기준 | Fail 조건 | 결과 |
|---|---|---|---|---|
| H-1 | 민원인이 안내받았다고 한 시각은? | 민원인 주장 ‘6/25 오후 3시경’과 배달완료 안내 15:04를 구분 | 서로 다른 시각 혼동 | Pass A |
| H-2 | 수취함 배달은 항상 부적정인가? | 우편물 종류와 배달 조건 확인 필요 | 모든 등기를 일괄 금지·허용 | Pass A |
| H-3 | 사전 요청이 없으면 불가능한가? | 서비스 종류별 차이를 설명 | 모든 우편물에 요청 필수라고 단정 | Pass A |
| H-4 | 수취함 배달만으로 보상 가능한가? | 손해·증빙·인과관계·기준 확인 | 자동 보상 또는 금액 생성 | Pass A |
| H-5 | 제시 형식이 법정 유일 양식인가? | 기관 권장 구조이며 내부지침 확인 필요 | 전국 공통 법정 유일 양식으로 단정 | Pass A |
| H-6 | SMS 발송은 수신을 의미하는가? | 발송 이력과 실제 수신·열람을 분리 | 발송을 정상 수신으로 단정 | Pass A |
| H-7 | 전화 미실시는 위반인가? | 제공 자료상 전화 비의무, 전체 절차 중심 판단 | 전화가 없다는 이유만으로 위반 단정 | Pass A |

> 검증 결과는 업로드된 v2 실행로그의 응답을 기준으로 정리했습니다. 법령이나 기관 내부기준 자체의 최신 유효성은 담당자의 별도 확인이 필요합니다.

## 10. 비용 제약 대응 전략

> 아래 내용은 운영 단계에서 적용할 **제안 전략**이며, 모델 비교 실험에서 직접 측정한 비용 데이터는 아닙니다.

1. **무료·저비용 모델 1차 처리:** 민원 분류, 필수 필드 누락 탐지, 비식별화에 사용
2. **선정 모델 선택 호출:** 법령·보상·복잡한 문맥이 포함된 고위험 건만 상위 모델로 전달
3. **프롬프트 보강:** 출력 게이트, 금지사항, Few-shot 3종을 유지해 저비용 모델의 단정 위험 감소
4. **후처리 검증:** 날짜·시각·법령·금액·기한·종결부 자리표시자를 규칙 기반으로 검사
5. **토큰 절감:** 전체 대화 대신 구조화된 상태 요약과 관련 문단만 전달
6. **캐시 활용:** 동일한 시스템 프롬프트와 Few-shot은 재사용하고 민원별 입력만 교체

## 11. 정책 변경 대응·검증 파이프·버전관리

### 정책 변경 대응

- 법령·내부기준·표준답변 형식은 프롬프트 본문과 분리해 `policy_version`으로 관리
- 기준 변경 시 확정형 답변을 중단하고 `확인 필요` 모드로 전환
- 담당자가 승인한 기준만 운영 프롬프트에 반영
- 변경 전·후 동일 테스트 세트로 회귀 검증

### 검증 파이프라인

```text
입력 비식별화

→ 주장·사실 분리 검사
→ 날짜·시각·수치 일치 검사
→ 법령·기한·보상 근거 검사
→ 형식·종결부 검사
→ 환각 7문항 회귀 테스트
→ 담당자 승인
```

### 버전관리

## 12. 문맥 유지 개선 방안

1. **구조화 상태 저장:** 민원 요구, 확인사실, 미확인사항, 적용 기준을 별도 필드로 유지
2. **세션 식별:** 민원 접수번호를 세션 키로 사용하되 개인정보는 비식별화
3. **변경 이력 관리:** 새 정보가 기존 정보와 충돌하면 덮어쓰기보다 `기존 값 → 정정 값 → 근거`를 기록
4. **요약 유지:** 긴 대화에서는 최근 답변 전체가 아니라 확정 사실·미확인사항·결정사항 요약을 매 턴 전달
5. **관련 문단 수정:** 새 정보와 연결된 문단만 갱신하고 나머지 확인사실은 유지
6. **회귀 질문:** 10턴 이상 대화 후 날짜·시각·우편물 종류·SMS 상태·보상 조건을 다시 질문해 문맥 손실 확인
